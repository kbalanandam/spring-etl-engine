# Hierarchical Flow Composition  ## Purpose  This note freezes the current working direction for reusable composition in OneFlow while the implementation matures.  The guiding principle is:  > reusable jobs and components evolve into scenario-driven execution  That means reuse remains primary, while one selected scenario becomes the explicit execution context that binds, orders, tracks, and evidences the reusable assets.  It also preserves one longer-term rule beyond the first implementation slice:  > a main flow must eventually be reusable as a subflow inside a higher-level flow  ## Status  - Classification: **Current baseline + future evolution** - This note is the current architectural reference for the intended bounded first slice of the `main flow -> subflow -> step` model and the longer-term direction toward reusable composition at any flow level.  ## Frozen working direction  One selected scenario may represent one full business flow.  That main flow may contain multiple reusable subflows.  Each subflow may contain one or more ordered executable steps.  That is the frozen first implementation slice.  The longer-term direction should generalize that model so reusable flow units can be embedded at higher levels. In that later shape, a main flow from one context may be reused as a subflow inside another main flow without changing the single-scenario execution boundary.  The intended hierarchy is therefore:  ```text Scenario execution   -> Main Flow       -> SubFlow           -> Step ```  Today the shipped baseline is still an explicit flat `job-config.yaml` `steps` list.  The frozen target shape is hierarchical in design but should remain explicit and predictable in execution.  For the first implementation slice, use the bounded hierarchy in this note. For the longer-term product direction, treat `main flow` and `subflow` as role labels within a reusable flow model rather than as permanently separate kinds of reusable assets.  For the practical rules that keep simple authored flows lightweight while still normalizing to one internal model, continue in [`flow-normalization-rules.md`](flow-normalization-rules.md).  ## Core model  ### 1. Scenario execution  The scenario remains the selected runtime boundary.  It is responsible for:  - selecting one runnable config bundle - binding the reusable flow structure to concrete config assets - carrying run identity, logging identity, and evidence identity - executing the resolved plan end to end  ### 2. Main Flow  A main flow is the top-level reusable business flow executed inside one selected scenario.  It should:  - act as the ignition/platform context for the run rather than as one more work task - represent the full business intent for the run - own the ordered list of subflows - produce the top-level flow summary for logging and evidence - provide the common context that spans all contained subflows and steps - carry shared control-plane artifacts such as run identity, logging context, recovery policy, and cross-subflow handshake metadata  In the first bounded slice, this is the top flow inside one selected scenario. Over time, the same reusable flow definition should be able to serve as a nested subflow inside a higher-level main flow.  ### MainFlow shared-context rule  `MainFlow` should be allowed to carry the shared context that one reusable work phase may need from another, but that context must stay disciplined.  Good `MainFlow` shared context includes:  - run and flow identity - recovery and execution policy - top-level logging/evidence context - cross-subflow handshake readiness - named subflow status visible to downstream reusable work phases - upstream-failure blocking / skip reasons for later subflows - artifact references, schema/version markers, checksums, counts, or compatibility flags  `MainFlow` should **not** become a dumping ground for the actual business payload. Large record sets, XML trees, or transformed business objects should continue to move through explicit step/subflow outputs and inputs.  ### 3. SubFlow  A subflow is a reusable grouped flow component within the main flow.  A subflow should:  - represent one meaningful business or technical phase - contain one or more ordered steps - own subflow-level logging, tracking, and summary evidence - remain reusable across multiple main flows where appropriate - consume and produce explicit work artifacts while using `MainFlow` shared context only for control-plane handshake metadata  Each subflow should also have:  - a stable reusable name distinct from `MainFlow` - an explicit execution status such as `READY`, `NOT_STARTED`, `COMPLETED`, `FAILED`, `BLOCKED`, or `SKIPPED`  That status should be published into `MainFlow` shared context so later subflows can consume it to decide whether to start, wait, skip, or stop.  For the bounded first descriptor slice, the expected default control rule is:  - a downstream subflow may start only after the required upstream subflow reaches `COMPLETED` - a downstream subflow is blocked if the required upstream subflow reaches `FAILED` or `BLOCKED` - if a downstream subflow consumes a handoff alias, that handoff must be marked ready before the downstream subflow is allowed to start  Those rules are currently descriptor-only metadata. They clarify orchestration intent now and can be enforced later when the runtime grows first-class subflow execution.  In the longer-term direction, a subflow is best understood as a reusable flow unit playing a nested role inside another flow. That means the platform should eventually allow one reusable flow to appear either as a main flow or as a subflow depending on the selected composition.  Examples:  - ingestion subflow - normalization subflow - validation subflow - export subflow  ### 4. Step  A step remains the smallest executable unit in the current runtime baseline.  Today that means one ordered:  - `source -> processor -> target`  A step should continue to be the place where detailed execution metrics are measured.  ## Execution rule  The model may be hierarchical, but the runtime should stay operationally predictable.  The preferred execution rule is:  > nested in design, explicit in execution  That means the engine may preserve the hierarchy for meaning and observability, but the selected scenario should still compile into one explicit execution plan.  For example:  ```text Main Flow   -> SubFlow A       -> Step 1       -> Step 2   -> SubFlow B       -> Step 3       -> Step 4 ```  may execute as one ordered scenario plan such as:  ```text SubFlow A / Step 1 SubFlow A / Step 2 SubFlow B / Step 3 SubFlow B / Step 4 ```  This keeps runtime behavior deterministic without losing the hierarchical flow model.  The same rule should continue to apply if reusable flow composition later becomes deeper than the first bounded slice: nested reusable flow structure may grow, but execution should still resolve into one explicit scenario plan.  ## Logging hierarchy  Logging should exist at three levels.  ### Main flow logging  This level should capture:  - scenario name - main flow name - run identifier - overall status - overall timing - rolled-up counters  ### Subflow logging  This level should capture:  - subflow name - subflow order - parent main flow identity - subflow status - subflow timing - subflow rolled-up counters  ### Step logging  This level should capture:  - step name - source - target - read/write/filter/reject counts - timing - output and reject artifacts where relevant  The intended hierarchy is:  ```text RUN_EVENT / MAIN_FLOW   -> SUBFLOW_EVENT / SUBFLOW_SUMMARY       -> STEP_EVENT / STEP_SUMMARY ```  One scenario log stream may carry all three levels as long as the context fields remain explicit.  ## Evidence hierarchy  Evidence should also roll up at three levels.  ### Main flow evidence  - final status - duration - rolled-up counts - primary business artifacts - overall verification summary  ### Subflow evidence  - subflow status - subflow duration - aggregated counts for contained steps - artifacts produced by that subflow - subflow warnings and failures  ### Step evidence  - detailed counts - output artifacts - reject artifacts - archive artifacts - step-level failures  The roll-up order should be:  ```text Step evidence -> Subflow summary -> Main flow summary ```  ## Reuse rule  Reuse remains primary in this direction.  That means:  - reusable components should not disappear behind scenario-specific code - subflows should remain reusable grouped assets - main flows should be reusable composed assets where practical - scenarios should select and execute those reusable assets explicitly  The scenario is therefore not the replacement for reuse.  It is the explicit execution context for reusable assets.  ## Guardrails  ### 1. Keep one normal execution boundary  One selected scenario remains the normal execution boundary.  Avoid defaulting to nested child jobs or separate runtime launchers for each subflow.  ### 2. Limit hierarchy depth  Freeze the model at three levels for now:  - main flow - subflow - step  Avoid recursive flow nesting in the first implementation slice until the initial hierarchical model is proven.  This is a temporary implementation constraint, not the final reuse principle. The long-term direction should still allow a reusable flow to be embedded as a nested flow in another composition once the bounded first slice is operationally proven.  ### 3. Preserve shared extension seams  Subflows and steps should continue to use shared readers, processors, writers, validation rules, and runtime support.  Do not grow one-off scenario-specific implementations just because the hierarchy is richer.  ### 4. Keep dependencies explicit  If one subflow or step depends on another, the relationship should be explicit in the selected scenario plan.  Do not rely on hidden inference.  ### 5. Keep observability hierarchical and uniform  Do not invent a different logging or evidence contract per subflow.  The hierarchy may differ, but the event model should stay uniform.  ## Relationship to the shipped baseline  The current shipped baseline is still:  - one selected `job-config.yaml` - one explicit ordered `steps` list - one scenario-aware run log - one scenario summary plus step-level evidence  That baseline remains valid.  This hierarchical flow model should evolve from it, not replace it with an unrelated orchestration system.  A practical bridge is:  - keep `job-config.yaml` `steps` as the executable baseline - allow future metadata to group those steps into subflows inside the same selected scenario - keep scenario-aware logging and evidence while extending them with main-flow and subflow identity  ## Diagram  ```mermaid flowchart TD     Scenario[Selected Scenario Execution]     MainFlow[Main Flow\nTop-level reusable business flow]      SubA[SubFlow A\nReusable grouped phase]     SubB[SubFlow B\nReusable grouped phase]      Step1[Step 1\nsource -> processor -> target]     Step2[Step 2\nsource -> processor -> target]     Step3[Step 3\nsource -> processor -> target]      LogMain[Main Flow Summary\nrun status + rolled-up evidence]     LogSub[SubFlow Summary\nphase status + rolled-up evidence]     LogStep[Step Evidence\ncounts + artifacts + failures]      Scenario --> MainFlow     MainFlow --> SubA     MainFlow --> SubB     SubA --> Step1     SubA --> Step2     SubB --> Step3      Step1 --> LogStep     Step2 --> LogStep     Step3 --> LogStep     LogStep --> LogSub     LogSub --> LogMain ```  ## Recommended terms  Use these terms consistently:  - **scenario execution** ΓÇö the explicit runtime selection and run context - **main flow** ΓÇö the top-level reusable business flow inside one selected execution in the bounded first slice - **subflow** ΓÇö a reusable grouped phase inside the main flow in the bounded first slice; longer term, this should be one reusable flow playing a nested role inside another flow - **step** ΓÇö the smallest executable `source -> processor -> target` unit in the current baseline  When discussing the longer-term model, it is acceptable to collapse `main flow` and `subflow` into the broader term **flow** so reuse at any level remains explicit.  ## Related docs  - [`Scenario-driven runtime direction`](scenario-driven-runtime-direction.md) - [`Flow normalization rules`](flow-normalization-rules.md) - [`Runtime flow`](runtime-flow.md) - [`Job history and operational observability`](../control-plane/job-history-and-operational-observability.md) - [`Job config reference`](../../config/job-config.md)
