# CSV Native Parser Sidecar Protocol  ## Purpose  This note defines a concrete **CSV-first sidecar protocol** for a future native parser worker that stays behind OneFlow's existing Java/Spring Batch reader seam.  Use it when evaluating how a future C/C++ parser could be integrated without replacing `DynamicReader`, `ItemReader` / `ItemStreamReader`, generated-model resolution, or downstream ETL-core behavior.  ## Status  - Classification: **Future direction** - This is a protocol sketch for future adoption readiness, not a shipped runtime path.  ## Why start with CSV first  If OneFlow ever proves a real need for native parsing, CSV is the safest first source family because:  - the current CSV parser contract is already narrow and source-native - record boundaries are simpler than nested XML flattening - malformed-row categorization is easier to isolate from business rules - the Java adapter can keep mapping and lifecycle responsibilities without inventing a second ETL core  This note does **not** open native XML parsing by default. XML can be revisited later only if preserved scenarios prove a parser-native gap that still fits the same boundary.  ## Design goal  Define a future sidecar protocol that allows a native CSV parser worker to:  - parse one configured CSV source artifact - emit source-native rows or parser-native row failures - return opaque checkpoint tokens for restart participation - remain replaceable behind a Java `ItemStreamReader`  while leaving Java responsible for:  - Spring Batch lifecycle - generated-model mapping - runtime categorization and evidence - processor, writer, and orchestration behavior  The concrete Java-side adapter responsibilities for this protocol are documented in [`Java native parser reader adapter contract`](java-native-parser-reader-adapter-contract.md).  ## Non-goals  This protocol is **not** for:  - processor transforms - processor-rule reject/accept decisions - duplicate handling - target routing or write policy - scheduler or control-plane orchestration - bypassing the selected-job runtime contract  ## Recommended runtime shape  ```mermaid flowchart LR     A[SourceConfig] --> B[CsvDynamicReader or future native CSV reader adapter]     B --> C[Java ItemStreamReader adapter]     C --> D[Native parser client]     D --> E[Native CSV parser sidecar]     E --> F[Parsed row / parser event stream]     F --> G[Java mapping into generated source record]     G --> H[Processor transforms]     H --> I[Processor rules]     I --> J[Writer] ```  ## Sidecar boundary rule  The sidecar must stop at **source-native row interpretation**.  It may emit:  - parsed field values in configured column order - parser-native malformed-row details - checkpoint tokens - end-of-stream markers  It must not emit:  - target-ready objects - transformed business values - rule-evaluated acceptance decisions - duplicate-resolution outcomes  ## Transport shape  The preferred first protocol is a **local sidecar process** launched by the Java reader adapter and communicating over a simple framed stream.  Acceptable future transports include:  - standard input/output with framed JSON lines - local TCP socket - named pipe - gRPC over loopback  The transport may vary later, but the message model should stay narrow and parser-native.  ## Message model  ### Open request  The Java adapter starts one parsing session with a request that contains only source-native parser inputs.  Suggested shape:  ```json {   "type": "open",   "sessionId": "run-2026-05-16-001",   "sourceName": "CustomerCsv",   "filePath": "C:/data/input/customers.csv",   "parser": {     "delimiter": ",",     "quoteCharacter": "\"",     "skipHeader": true,     "columnNames": ["customerId", "customerName", "countryCode"],     "strictColumnCount": true   },   "checkpointToken": null } ```  ### Parsed row event  The sidecar returns one source-native row at a time.  ```json {   "type": "row",   "rowNumber": 42,   "values": {     "customerId": "C10042",     "customerName": "Acme Ltd",     "countryCode": "USA"   },   "checkpointToken": "offset=184449,row=42" } ```  ### Malformed-row parser event  This event reports parser-native failure before a normal runtime record exists.  ```json {   "type": "malformed_row",   "rowNumber": 43,   "reasonCode": "UNTERMINATED_QUOTE",   "message": "Quoted field was not terminated before end of row.",   "rawRowPreview": "C10043,\"Broken Customer",   "checkpointToken": "offset=184611,row=43" } ```  ### End event  ```json {   "type": "end",   "finalCheckpointToken": "offset=984221,row=10000" } ```  ### Fatal parser error event  ```json {   "type": "fatal_error",   "reasonCode": "FILE_OPEN_FAILED",   "message": "Unable to open source file.",   "checkpointToken": null } ```  ## Java adapter lifecycle mapping  The Java reader adapter should still implement Spring Batch lifecycle explicitly.  ### `open(...)`  Java adapter responsibilities:  - validate the selected source config - start or connect to the native sidecar - send the `open` request - restore any stored checkpoint token from `ExecutionContext`  ### `read()`  Java adapter responsibilities:  - read one sidecar event - map `row` events into the generated source model class - convert `malformed_row` or `fatal_error` events into the OneFlow runtime failure surface - return one item per call when a normal runtime record exists  ### `update(...)`  Java adapter responsibilities:  - persist the latest opaque `checkpointToken` into Spring Batch `ExecutionContext` - avoid interpreting native parser internals beyond storing the token  ### `close()`  Java adapter responsibilities:  - close the sidecar client/session - terminate the sidecar process if OneFlow launched it - release temporary resources  ## Checkpointing rule  The checkpoint token should be treated as **opaque to the ETL core**.  Examples of acceptable internal meaning on the native side:  - byte offset - row number - parser engine cursor state - fragment boundary token  Examples of what Java should do:  - store the token - resend it on reopen - log that a checkpoint exists when helpful for diagnostics  Examples of what Java should not do:  - parse the token into business logic - derive orchestration behavior from native parser state - depend on native memory layout details  ## Error handling rule  The sidecar protocol may expose parser-native reason codes, but Java remains responsible for the runtime-facing contract.  That means:  - sidecar emits parser-native events and reason codes - Java adapter maps them into OneFlow runtime categories and exceptions - operator-visible logs and evidence stay aligned with the existing Java runtime model  ## Ownership split  ### Native sidecar owns  - CSV tokenization and quote/escape handling - row-boundary detection - parser-native malformed-row detection - checkpoint token generation  ### Java adapter owns  - Spring Batch `ItemStreamReader` lifecycle - generated-source-record creation - interaction with `RuntimeCategorizingItemStreamReader` or equivalent runtime wrapper - handoff into processor transforms and rules - final ETL-core behavior after record creation  ## Rollout guidance  If this direction ever becomes active, keep the rollout narrow:  1. one CSV source family only 2. one local sidecar transport 3. one opaque checkpoint token contract 4. one Java reader adapter that preserves the current runtime boundary 5. preserved scenario proof before broadening native-parser scope  ## Related docs  - [`native-parser-adoptability.md`](native-parser-adoptability.md) - [`java-native-parser-reader-adapter-contract.md`](java-native-parser-reader-adapter-contract.md) - [`oneflow-file-parser-capabilities.md`](oneflow-file-parser-capabilities.md) - [`extension-points.md`](extension-points.md)  ## Bottom line  If OneFlow ever adopts a native parser engine, a **CSV-first sidecar protocol** is the safest first concrete shape.  It keeps native parsing powerful where it matters, but still preserves the core rule:  **Java owns the reader/runtime contract; native code owns source-native parsing only.**
