# Java Native Parser Reader Adapter Contract  ## Purpose  This note defines the **Java-side adapter contract** for a future native parser integration in `spring-etl-engine`.  Use it when designing a Java `DynamicReader` / `ItemStreamReader` implementation that talks to a native sidecar or worker while preserving the current Spring Batch runtime seam.  ## Status  - Classification: **Future direction** - This note describes the preferred adapter shape for future native parser adoption; it is not a shipped reader path.  ## Why this note exists  The native-parser notes already preserve two high-level rules:  - native parser engines must stay behind the Java reader boundary - sidecar-style integration is preferred over deep in-process coupling by default  What still needs to stay concrete is the **Java adapter itself**. Without that contract, a future native parser effort could still drift into a parser-centered design where lifecycle, mapping, error handling, or checkpoint semantics leak out of Spring Batch and into ad hoc transport code.  This note freezes the opposite rule: the Java adapter remains the runtime-facing owner even when parsing happens elsewhere.  ## Stable Java seam to preserve  The future adapter must fit the existing active runtime shape:  ```mermaid flowchart LR     A[SourceConfig] --> B[DynamicReaderFactory]     B --> C[DynamicReader]     C --> D[Java ItemStreamReader adapter]     D --> E[Native parser client / sidecar]     E --> F[Source-native row or parser event]     F --> G[Generated runtime source record]     G --> H[Processor transforms]     H --> I[Processor rules]     I --> J[Writer] ```  Read that diagram with one strict rule:  - the adapter is the runtime-facing seam - the native parser is only the parser engine behind that seam  ## Current code anchors  Keep future adapter work aligned with these active runtime anchors:  - `src/main/java/com/etl/reader/DynamicReader.java` - `src/main/java/com/etl/reader/DynamicReaderFactory.java` - `src/main/java/com/etl/reader/impl/RuntimeCategorizingItemStreamReader.java` - `src/main/java/com/etl/reader/mapper/DynamicFieldSetMapper.java` - `src/main/java/com/etl/common/util/GeneratedModelClassResolver.java`  ## Adapter responsibilities  A future Java native-parser adapter should own:  - `SourceConfig` interpretation for the selected source format - Spring Batch `ItemStreamReader` lifecycle participation - client/session creation for the native parser sidecar - conversion from source-native parser events into generated Java source records - storage and replay of opaque checkpoint tokens through `ExecutionContext` - translation of parser-native failures into the OneFlow runtime-facing exception surface  The adapter must **not** own:  - processor transforms - processor-rule accept/reject business decisions - duplicate policy - target routing or publication behavior  ## Recommended class shape  A future implementation should still enter through a normal `DynamicReader`.  Recommended conceptual split:  - format-specific `DynamicReader` implementation - format-specific Java adapter that implements `ItemStreamReader<T>` - parser client that speaks the sidecar protocol - row-mapper that converts parser values into the generated source class  Example conceptual layout:  ```text NativeCsvDynamicReader   -> NativeCsvItemStreamReader<T>       -> NativeParserClient       -> NativeParserRowMapper<T> ```  ## Lifecycle contract  ### `getReader(...)`  The `DynamicReader` implementation should:  - validate that the selected `SourceConfig` type is supported - resolve the generated source model class as today - build the Java adapter, not a parser-centered runtime object - return a normal Spring Batch `ItemReader` / `ItemStreamReader`  ### `open(ExecutionContext)`  The adapter should:  - validate parser-native config inputs - open or connect to the native sidecar session - restore the last stored opaque checkpoint token if one exists - initialize any adapter-local buffers needed for one-item-per-read behavior  The adapter should **not**:  - infer business restart semantics from native parser internals - let native engine state escape as public runtime contract details  ### `read()`  The adapter should:  - request or receive one parser event from the native sidecar - handle event types explicitly - return **at most one generated runtime source record per call** - preserve Spring Batch's normal end-of-stream contract by returning `null` only when the sidecar indicates the stream is finished  Recommended event handling:  - `row` -> map to generated source record and return it - `malformed_row` -> convert to OneFlow runtime failure surface immediately by default - `fatal_error` -> raise a categorized runtime failure immediately - `end` -> return `null`  ### `update(ExecutionContext)`  The adapter should:  - persist the most recent opaque checkpoint token - persist only minimal adapter-local state needed to restart safely - avoid making `ExecutionContext` dependent on native memory layouts, parser object handles, or transport-specific internals  ### `close()`  The adapter should:  - close the parser client/session - clear local buffers - release adapter-owned resources - tolerate already-ended sessions when possible  ## Checkpoint token rule  Checkpoint tokens should stay **opaque** to the ETL core.  That means:  - the native parser may generate the token - the Java adapter stores it - the Java adapter replays it on reopen - the token should not be parsed into orchestration or business logic  Acceptable examples:  - byte offset token - row-number token - parser-cursor token  Unacceptable examples:  - a token whose interpretation changes processor or writer behavior - a token that forces the ETL core to understand native parser buffer layout  ## Row mapping contract  The adapter should convert parser-native values into the generated source model using a Java-side mapping path consistent with the current reader model.  Preserve these rules:  - configured field names still define the expected generated writable properties - Java still owns type conversion into the generated source record - mapping failures still surface as runtime reader failures  In other words, the native parser may supply field values, but Java remains the owner of the final generated-model handoff. That keeps the future adapter aligned with the spirit of `DynamicFieldSetMapper` instead of moving model-binding responsibility into native code.  ## Malformed-row handling rule  Recommended default:  - treat parser-native malformed-row events as **reader/runtime failures** unless a future parser-specific reject policy is explicitly designed and documented  Why this default is safer:  - it preserves the current runtime boundary - it avoids inventing a second reject/quarantine path inside the parser layer - it keeps parser-native failure handling visible and fail-fast until a narrower alternative is justified  If a future design ever introduces tolerate-and-report behavior for malformed parser rows, document it as an explicit parser/runtime contract change first.  ## Error mapping rule  The adapter remains responsible for converting parser-native failures into the OneFlow runtime-facing exception model.  That means:  - parser-native reason codes are allowed internally - operator-facing exception shape stays Java-owned - runtime evidence and categorization stay aligned with the rest of the Spring Batch reader path  A future adapter should prefer wrapping or aligning with the same runtime-failure conventions already preserved by `RuntimeCategorizingItemStreamReader`.  ## Buffering rule  If the sidecar protocol ever emits more than one logical row from one native parser event, the Java adapter may buffer those rows internally.  But preserve this contract:  - downstream Spring Batch path still sees one item per `read()` - buffering remains an adapter concern, not a processor or orchestration concern  This is the same general discipline used by fragment-buffering XML reader adapters today.  ## Transport isolation rule  The adapter should depend on a narrow parser client abstraction, not on transport-specific details spread through reader logic.  Keep transport concerns behind one small client boundary such as:  - `openSession(...)` - `readEvent()` - `checkpointToken()` or equivalent - `closeSession()`  That preserves the ability to change between:  - stdio - loopback socket - named pipe - gRPC  without redesigning the reader contract.  ## Decision checklist for future implementation  A proposed Java adapter is aligned only if all of these stay true:  1. it is still returned through `DynamicReader.getReader(...)` 2. it still behaves as a normal Spring Batch `ItemReader` / `ItemStreamReader` 3. it still hands off a generated runtime source record to the ETL core 4. it keeps checkpoint tokens opaque 5. it keeps malformed parser failures on the Java runtime-failure surface unless a new contract is explicitly documented 6. it keeps transport mechanics outside processor, writer, and orchestration code  ## Related docs  - [`native-parser-adoptability.md`](native-parser-adoptability.md) - [`csv-native-parser-sidecar-protocol.md`](csv-native-parser-sidecar-protocol.md) - [`oneflow-file-parser-capabilities.md`](oneflow-file-parser-capabilities.md) - [`../product/backlog-items/P5-native-parser-adoptability-and-sidecar-integration-readiness.md`](../../product/backlog-items/P5-native-parser-adoptability-and-sidecar-integration-readiness.md)  ## Bottom line  If OneFlow ever adopts a native parser, the **Java adapter must remain the runtime-facing owner**.  The adapter is where Spring Batch lifecycle, generated-model mapping, checkpoint persistence, and runtime error alignment stay stable.  The native parser may parse source-native structure, but it must not replace the Java reader contract.
