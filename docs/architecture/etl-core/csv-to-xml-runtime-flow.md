# CSV to XML Runtime Flow  ## Purpose  This page explains the shipped runtime path for **CSV source -> XML target** execution as an operational flow, not as a per-class catalog.  Use it when you want to understand:  - which runtime components participate in a CSV-to-XML run - how the selected job bundle becomes a concrete Spring Batch step plan - where flat XML and nested XML target behavior diverge - how validation, reject handling, duplicate handling, archive-on-success, and staged publication fit into the same runtime path - which logs and preserved bundles provide proof for the behavior  For the broader one-run baseline across all formats, start with [`runtime-flow.md`](runtime-flow.md).  ## Status  - Classification: **Current runtime** - Scope: **CSV source to XML target runtime behavior on the shipped path**  ## Preserved bundle anchors  This note uses the following checked-in bundles as the main proof anchors:  - [`src/main/resources/config-jobs/customer-load/README.md`](../../../src/main/resources/config-jobs/customer-load/README.md) ΓÇö simplest shipped `CSV -> XML` path writing a final XML output - [`src/main/resources/config-jobs/csv-to-nested-xml/README.md`](../../../src/main/resources/config-jobs/csv-to-nested-xml/README.md) ΓÇö advanced shipped `CSV -> nested XML` path using `modelDefinitionPath` - [`src/main/resources/config-jobs/csv-validation-reject-archive/README.md`](../../../src/main/resources/config-jobs/csv-validation-reject-archive/README.md) ΓÇö shipped proof for CSV-ingestion reject handling and archive-on-success on the same active CSV file-ingestion path  The product does **not** currently rely on one single preserved bundle to prove every optional hardening feature together with XML output. Instead, the shipped CSV-to-XML runtime shape shares the same CSV ingestion, processor, duplicate, reject, archive, and staged-publication seams proven across those preserved bundles.  ## Flow summary  At a high level, a CSV-to-XML run works like this:  1. `ConfigLoader` selects one explicit `job-config.yaml` bundle 2. `EtlJobRunner` seeds run parameters and logging context 3. `BatchConfig` resolves the selected step's source, target, processor mapping, and generated model metadata 4. `DynamicReaderFactory` chooses the CSV reader path 5. `DynamicProcessorFactory` chooses the shared processor path 6. `GeneratedModelClassResolver` resolves XML processing/write classes and wrapper metadata 7. `DynamicWriterFactory` chooses the XML writer path 8. Spring Batch runs the step in chunk or tasklet mode 9. the XML target is published only after staged-write rules are satisfied 10. listeners emit run, step, subflow, reject, and archive evidence  ## 1. `job-config.yaml` Controls:  - selected bundle identity and log scenario name - explicit source/target/processor file references - explicit ordered `steps[]`  ### 2. `source-config.yaml` (CSV) Controls:  - CSV input file path - delimiter and field list - optional archive-on-success behavior - generated source package/class resolution inputs  ### 3. `processor-config.yaml` Controls:  - shared processor contract (`type: default`) - field mappings from CSV source fields into XML target fields - optional transforms - optional processor rules - optional reject handling  For duplicate handling, the active shipped config shape is still processor-rule driven:  - `rules[].type: duplicate` enables duplicate detection for the mapped field - `rules[].keyFields` optionally defines the logical duplicate key - `rules[].orderBy` upgrades duplicate handling from keep-first mode into ordered winner selection For ordered winner selection (`duplicate + orderBy`), the processor config can optionally set `storageMode: memory|embeddedDb`; when omitted (or set to `auto`), runtime selects the resolver implementation from step volume:  - smaller candidate sets stay on the in-memory resolver path - larger or unknown-size candidate sets switch to the embedded-database resolver path  Terminology note: processor config uses `memory`, while runtime resolver evidence emits `inMemory`.  See also: [`Default processor reference`](../../config/processor/default-processor.md), [`File ingestion hardening`](file-ingestion-hardening.md), and [`T4 ΓÇö Transformation quarantine and duplicate hardening`](../../product/backlog-items/T4-transformation-quarantine-and-duplicate-hardening.md).  ### 4. `target-config.yaml` (XML) Controls:  - XML target file path - `rootElement` - `recordElement` - optional `modelDefinitionPath` for nested XML targets - generated target processing/write class resolution inputs  ## Flat XML vs nested XML target shape  The CSV-to-XML runtime path splits into two important XML target shapes.  ### Flat XML target This is the simpler shape used by `customer-load`:  - one CSV row maps to one XML record object - the XML target uses the normal top-level XML metadata fields - final output is written through the standard XML writer path  ### Nested XML target This is the advanced shape used by `csv-to-nested-xml`:  - one CSV row still starts as one source record - processor mappings may target nested XML paths such as `profile.email` or `address.city` - the target keeps the same top-level XML config shape, but adds `modelDefinitionPath` - generated nested XML model classes come from that structural definition - final publication still flows through the same XML writer seam  Operationally, this means the XML target contract remains stable while the generated target model shape becomes richer.  ## Generated model resolution in this flow  `GeneratedModelClassResolver` is the key compatibility seam for CSV-to-XML.  For a CSV source to XML target step, it resolves:  - source class name for the generated CSV row model - target processing class name for record-level mapping and chunk processing - target write class name for final XML write/publication - XML wrapper field metadata when the XML output needs a wrapper/root object  This distinction matters because XML targets may use:  - **processing class** ΓÇö one logical XML record item - **write class** ΓÇö wrapper/root object used to publish the final XML document  That split is what lets the same runtime support both chunk-oriented XML record output and tasklet/wrapper-style final document publication.  ## Chunk vs tasklet behavior on CSV -> XML  `BatchConfig` chooses chunk or tasklet execution from record count plus duplicate-handling rules.  ### Chunk path Typical when:  - record count exceeds `etl.chunk.threshold` - no ordered duplicate winner-selection override is active  Operationally:  - CSV reader streams records - processor maps each accepted record into the XML processing class - XML writer streams XML records to staged output  ### Tasklet path Typical when:  - small CSV input falls below the threshold - ordered duplicate winner selection forces tasklet mode - wrapper-style XML publication is required at the end of buffered processing  Operationally:  - tasklet loops through the reader directly - accepted records are buffered in memory - when needed, the runtime creates one XML wrapper/root object - one final XML document is written through the wrapper path  ## Validation, reject handling, and duplicate behavior  The shipped processor order on the CSV-to-XML path is:  `read -> transforms -> processor rules -> write`  ### Transform behavior Transforms clean and normalize mapped values before rule evaluation.  ### Rule behavior Rules can:  - reject a record and continue when `onFailure=rejectRecord` - fail the step immediately when configured to fail the step  ### Reject handling When reject handling is enabled:  - rejected rows are written to a reject CSV artifact - step evidence includes `rejectedCount` - step evidence may include `rejectOutputPath`  ### Duplicate behavior Two shipped duplicate modes exist:  - keep-first / reject-later duplicate detection on the normal step path - ordered winner selection, which upgrades the step to tasklet mode so final winners are resolved before final write  For ordered winner selection, there is one more internal runtime split worth knowing operationally:  - if the step's record count is below the active chunk threshold, the runtime keeps duplicate   winner selection in memory - if the record count is above the threshold, or the count cannot be determined and the runtime   defaults to the large-input path, the runtime uses the embedded-database duplicate resolver  That means the user-facing config expresses **duplicate semantics** through `duplicate`, `keyFields`, and `orderBy`, while the runtime chooses the storage strategy automatically from step volume.  ## Archive-on-success behavior for CSV source  When the selected CSV source enables archive-on-success:  - archive processing happens only after the step completes successfully - the original processed CSV source is moved or renamed into the configured archive location - step evidence may include `archivedSourcePath`  This is proven by the shipped CSV hardening slice and applies to CSV->XML because the same CSV ingestion runtime support is reused.  ## XML publication behavior  The XML target path publishes through staged output.  ### Staged write rule The XML output is written to a sibling `.part` file first.  ### Publish rule Final publication happens only after:  1. the writer stream closes cleanly 2. the enclosing step completes successfully  ### Failure rule If the step fails or the writer fails:  - the staged XML artifact is cleaned up - incomplete XML is not promoted as final output  ### Wrapper-mode note For wrapper/root-object publication:  - the runtime marshals one logical wrapper object as the full final XML document - if a completed wrapper-mode step emitted no wrapper object, stale published output may be removed rather than left behind as if fresh output was produced  ## Operational evidence for CSV -> XML  ### Startup-time evidence In `logs/startup/startup.log`:  - `STEP_PLAN` - `STEP_READY`  ### Scenario-scoped evidence In `logs/<yyyy-MM-dd>/<scenario>.log`:  - `RUN_EVENT` - `MAIN_FLOW_PLAN` - `SUBFLOW_PLAN` - `STEP_EVENT` - `SUBFLOW_SUMMARY` - `RUN_SUMMARY`  ### Evidence fields especially relevant to CSV -> XML Watch for:  - `stepName` - `source` / `target` - `mode` (`chunk` or tasklet-style execution) - `rejectedCount` - `rejectOutputPath` - `archivedSourcePath` - `writtenCount` - `handoffWriteCount` when the XML step is intermediate rather than final  ## Failure surfaces and what they usually mean  | Symptom | Likely layer | |---|---| | startup fails before job launch | `ConfigLoader` selected config validation | | XML classes not found | generated model resolution / generation mismatch | | CSV rows rejected instead of written | processor rule evaluation / reject handling | | XML output missing but `.part` exists briefly | staged publication prevented promotion due to step or writer failure | | nested XML fields missing | processor mapping paths or nested XML target structure mismatch | | chunk unexpectedly became tasklet | record-count threshold or ordered duplicate winner-selection override |  ## How to read the shipped preserved examples together  ### `customer-load` Use this when you want to understand the simplest shipped `CSV -> XML` baseline:  - one selected CSV source - one XML target - simple field-to-field mapping - final XML output  ### `csv-to-nested-xml` Use this when you want the advanced XML target branch:  - flat CSV source - nested XML output - `modelDefinitionPath` - generated nested target model  ### `csv-validation-reject-archive` Use this when you want to understand the optional CSV ingestion hardening behavior reused by CSV->XML:  - reject output - validation-driven record rejection - archive-on-success - file-ingestion runtime support evidence  ## Operational checklist for a CSV -> XML run  Before running:  - confirm `etl.config.job` points to the intended `config-jobs/.../job-config.yaml` - confirm the CSV input exists at the configured runtime path - confirm processor mappings align with source field names and XML target field names - for nested XML, confirm the `modelDefinitionPath` file exists and matches intended output shape - if generated XML classes are job-scoped, confirm generation has already run when required  After running:  - check `logs/startup/startup.log` for `STEP_PLAN` and `STEP_READY` - check the scenario log for `STEP_EVENT`, `RUN_SUMMARY`, and any reject/archive evidence - confirm final XML output exists instead of only a transient `.part` file - if reject handling is enabled, confirm reject artifact location and `rejectedCount` - if archive-on-success is enabled, confirm `archivedSourcePath`  ## Related documents  -
