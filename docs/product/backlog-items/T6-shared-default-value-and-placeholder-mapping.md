## Summary

Define one explicit config contract for common target-field defaults so jobs can fill audit and operational columns without repeating the same field-by-field logic in every mapping. The first focus is shared defaults for columns such as `createdUser`, `updatedUser`, `createdDate`, and `updatedDate`, where audit date/date-time values come from one shared runtime system-date/date-time provider rather than from source data, plus a reusable path for fixed values and formula-derived runtime values.

## Problem

The shipped processor contract already supports direct field mapping, `valueMap`, and `expression` transforms, but shared defaulting still has no first-class product contract:

- audit fields such as `createdUser` / `updatedUser` and `createdDate` / `updatedDate` still need repetitive field-by-field authoring
- there is no shared placeholder contract for common runtime values such as selected job name, fixed constants, or a standard current timestamp value
- config authors must express simple defaulting intent through low-level per-field transform logic even when the real need is only â€śfill the usual audit columnsâ€ť
- future relational jobs will keep reintroducing the same mapping noise unless this becomes a first-class product pattern

That is especially visible in file-to-relational and multi-step jobs where source data does not naturally carry all required target audit columns.

## Goal

Add one explicit config pattern for shared default-value assignment so the product can fill common audit and operational columns predictably, while still allowing scenario authors to set fixed values or derive runtime values for other target fields when needed.

## Scope

This item covers:

- shared defaulting for common audit columns such as `createdUser`, `updatedUser`, `createdDate`, and `updatedDate`
- one shared system-date/date-time provider path for audit fields that must be generated by runtime instead of read from source payloads
- support for filling selected fields from the chosen job name or another explicit constant
- one product-defined default â€ścurrent timestampâ€ť behavior for generated values, including one standard rendered format where string/date formatting is needed
- a placeholder or formula-based path for other fixed values and derived runtime values
- validation guardrails so defaulted fields do not conflict with normal `from -> to` mapping or ordered `transforms[]`
- documentation and example-job updates once implementation begins

## Out of scope

This item does not cover:

- a generic rules engine or broad scripting platform
- secret injection or environment-secret management; keep that under `G1`
- database-trigger-only or database-default-only solutions as the sole product contract
- lookup/enrichment joins against external reference data; keep that under `T5`
- changing the selected-job runtime model or moving transformation behavior out of the processor seam by default

## Proposed approach

The preferred direction is:

1. keep the active processor transform path as the main extension seam instead of adding hidden target-writer-specific default injection
2. introduce an additive mapping-scoped default-value contract for target fields that need generated or shared values rather than source-derived input
3. provide short standardized runtime placeholders for common values such as selected job name, explicit constants, and a shared `currentDate` / `currentTimestamp` token backed by one runtime system-date/date-time provider
4. preserve `expression` for richer formulas, but avoid forcing every common audit column to be written as custom SpEL in each job
5. fail fast when the same field is assigned ambiguously through both normal source mapping and a default/placeholder rule
6. document one common default timestamp policy so users do not need to keep authoring format details for standard audit columns

### Illustrative future config shape (draft only)

The exact syntax is still backlog work, but one likely direction is a mapping-scoped `defaultValues:` block beside `fields:` so shared generated values stay explicit and separate from normal source-field mapping.

Illustrative example only â€” not a shipped contract yet:

```yaml
type: default
mappings:
  - source: OrdersCsvIntermediate
    target: OrdersSql
    fields:
      - from: orderId
        to: orderId
      - from: customerCode
        to: customerCode
      - from: amount
        to: amount
    defaultValues:
      - to: createdUser
        valueFrom: jobName
      - to: updatedUser
        valueFrom: constant
        value: OneFlow
      - to: createdDate
        valueFrom: currentTimestamp
      - to: updatedDate
        valueFrom: currentTimestamp
      - to: sourceSystem
        valueFrom: constant
        value: TVL
      - to: loadBatchLabel
        valueFrom: expression
        expression: "#jobName + '-' + #stepName"
```

This draft shape is meant to show three distinct needs:

- **shared audit defaults** â€” `createdUser`, `updatedUser`, `createdDate`, `updatedDate`
- **fixed operational constants** â€” values such as `sourceSystem: TVL`
- **derived runtime values** â€” fields such as `loadBatchLabel` built from job/runtime metadata through an explicit formula path

Important guardrails for the future contract:

- a target field should be assigned by only one path: normal `fields[]` mapping or `defaultValues[]`, not both
- common runtime values such as job name or current timestamp should have short standardized tokens instead of repeated custom expressions
- audit date/date-time defaults such as `createdDate` and `updatedDate` should resolve through one shared runtime provider/utility so jobs do not invent their own clock logic
- richer formulas can still reuse the existing expression direction, but only where simple placeholders are not enough

## Operator / runtime impact

Expected impact when this item ships:

- relational-target configs become smaller and more consistent across jobs
- common audit-column behavior becomes explicit instead of being hidden in ad hoc field mappings
- startup validation can point clearly to conflicting or unsupported placeholder/default definitions
- operators get more predictable metadata population across jobs without custom Java code
- docs can show one standard way to fill technical columns that are not present in the business source payload

## Acceptance criteria

- [ ] the product defines one documented contract for shared default-value mapping on top of the active processor transform seam
- [ ] common audit-column defaults can be configured without repeating custom expression logic in every job bundle
- [ ] selected job metadata and explicit constants can be used as supported generated values for target fields
- [ ] the product defines one shared runtime provider path for generated audit date/date-time values plus one default current-date/current-timestamp behavior and standard rendered format for the common audit use case
- [ ] config validation fails fast for unknown placeholders, ambiguous double assignment, and incompatible field/default combinations
- [ ] at least one relational-style preserved or private job shows the intended pattern when implementation begins

The important product guardrail is to keep this explicit and config-driven. Audit/default filling should not become a hidden writer-side behavior that differs by target type.

The first delivery slice should also define one shared runtime date/time provider or utility that becomes the source of truth for generated audit date/date-time defaults such as `createdDate` and `updatedDate`.

That same delivery slice should also decide and document what â€śserver timestampâ€ť means for the product contract: application-runtime generated current time, target-database generated value, or an explicitly selectable mode. Until that is decided, the backlog item should stay focused on contract direction and validation guardrails rather than on a final fixed syntax.

If the eventual syntax uses a mapping-level `defaultValues:` block similar to the draft above, the implementation should still resolve through the same processor-side transformation flow rather than introducing a separate hidden writer-default path.
