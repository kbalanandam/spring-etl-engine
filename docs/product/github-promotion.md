# GitHub Promotion Guide

## Purpose

This note is the in-repo source of truth for GitHub-facing product wording for **OneFlow**.

Use it to keep the repository presentation consistent across:

- GitHub About text
- README hero/tagline
- repository topics
- future social-preview copy

The goal is to present OneFlow clearly as a focused runtime for repeatable integration flows.

## Positioning guardrails

Use these rules in GitHub-facing copy:

- present **OneFlow** as the product-facing name
- keep `spring-etl-engine` as the stable technical repository identity
- describe OneFlow as a focused, config-driven runtime
- emphasize repeatable file-based and integration-oriented flows
- emphasize orchestration, validation, optional duplicate handling, reject/archive behavior, and operator-visible evidence
- keep the wording practical and believable

Avoid these patterns in GitHub-facing copy:

- direct comparison claims against Informatica, SSIS, or other traditional ETL suites
- wording that implies full traditional ETL-suite parity
- wording that implies OneFlow is already a broad enterprise middleware platform
- wording that implies OCR, image review, AI assistants, or unrelated learning topics are part of the current product promise unless explicitly added later

## Approved About copy

### Primary option

Focused config-driven runtime for repeatable file-based integration flows.

### Secondary option

Config-driven runtime for repeatable business and file-based integration flows.

### Slightly more technical option

Config-driven runtime for repeatable ETL and file-based integration scenarios.

## Approved tagline options

### Preferred tagline

Repeatable file flows, one runtime.

### Alternatives

- One runtime for repeatable integration flows.
- Config-driven flows without repeated ETL plumbing.
- One runtime for repeatable business flows.

## README alignment rules

The README opening should communicate these points in order:

1. `spring-etl-engine` is the technical identity
2. **OneFlow** is the product-facing name
3. the product is a focused runtime, not a traditional ETL suite
4. the current value is repeatable file-based and integration-oriented execution

## Messaging themes to repeat

Good recurring themes:

- focused runtime
- repeatable flows
- config-driven execution
- reduced repetitive ETL plumbing
- explicit orchestration
- operator-visible evidence
- practical delivery over platform sprawl

## Messaging themes to avoid

Avoid centering GitHub promotion on:

- full enterprise ETL breadth
- marketplace-style feature parity claims
- unrelated future ambitions
- experimental adjacent systems that are not part of the current product promise
